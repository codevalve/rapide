1. Overview

This document defines the requirements for adding Model Context Protocol (MCP) support to Rapide.

The goal is to allow local AI tools to interact with Rapide as a lightweight rapid-log journal backend, enabling AI agents to capture notes and events during development workflows.

The MCP implementation is intentionally narrow for the initial version to preserve Rapide’s simplicity, privacy, and local-first philosophy.

This feature will be released as an experimental capability and is not intended to expand Rapide into a generalized automation platform.

⸻

2. Goals

2.1 Primary Goal

Enable local AI tools to:
	1.	Add rapid-log entries to Rapide
	2.	Search AI-created entries
	3.	Retrieve recent AI-created entries for context

2.2 Developer Workflow Goal

During development sessions, an AI tool should be able to capture items such as:
	•	implementation notes
	•	observations
	•	TODOs
	•	small discoveries
	•	debugging notes

These entries should appear in Rapide so the developer can review them later as part of their normal journaling workflow.

⸻

3. Non-Goals (MVP)

The following capabilities are explicitly out of scope for the first implementation:
	•	remote or network MCP servers
	•	HTTP / WebSocket / SSE transports
	•	access to collections other than AGENT
	•	editing existing entries
	•	deleting entries
	•	pinning entries
	•	setting priority
	•	choosing bullet types
	•	tags or additional metadata
	•	prompts or resources beyond basic tool support
	•	authentication or user identity management
	•	cloud or multi-user deployment
	•	arbitrary filesystem access

These may be evaluated in later iterations if necessary.

⸻

4. Core Design Decisions

4.1 Local-Only Operation

The MCP server will run locally and communicate through stdio.

This ensures:
	•	privacy of journal data
	•	simplicity of deployment
	•	compatibility with local AI tooling
	•	minimal attack surface

The MCP server will not expose any network listeners.

⸻

4.2 Dedicated AI Collection

All AI-created entries are written to the collection:

margin_key = "AGENT"

This ensures AI activity remains separated from:
	•	personal journal entries
	•	work collections
	•	other user-created collections

The user may manually move or edit entries later within Rapide if desired.

⸻

4.3 Entry Type Restrictions

AI-created entries are constrained to standard rapid-log bullets.

The MCP server will enforce the following values:

bullet   = "•"
priority = false
pinned   = false

This prevents AI tools from altering entry importance or visibility.

Users may manually adjust these attributes later in Rapide.

⸻

4.4 Search Scope

Search operations must only access entries where:

margin_key == "AGENT"

This ensures AI tools cannot read personal or work-related notes from other collections.

⸻

4.5 Search Behavior

Search will use case-insensitive substring matching against the content field.

Example:

query: "scroll"

Matches:

Investigate scroll jitter in TUI
Scrolling bug discovered in table view

Search does not consider:
	•	bullet type
	•	timestamps
	•	priority
	•	pinned status

⸻

5. User Interface

5.1 Command

The MCP server will be launched via:

rapide mcp start

5.2 Feature Visibility

The command should be:
	•	hidden
	•	experimental
	•	minimally documented initially

This allows safe real-world experimentation before committing to a permanent interface.

⸻

6. MCP Tool Interface

The MVP implementation will expose three tools.

⸻

6.1 Tool: add_entry

Purpose

Append a new rapid-log entry to the AGENT collection.

Input

{
  "content": "Investigate TUI redraw jitter"
}

Validation

Rules:
	•	content is required
	•	content must not be empty after trimming

Server Controlled Fields

The MCP server will automatically set:

margin_key = "AGENT"
bullet = "•"
priority = false
pinned = false
timestamp = generated by Rapide
id = generated by Rapide

Example Stored Entry

{
  "id": "c421",
  "timestamp": "2026-03-11T10:21:13-05:00",
  "margin_key": "AGENT",
  "bullet": "•",
  "content": "Investigate TUI redraw jitter",
  "priority": false,
  "pinned": false
}

Response

Returns the created entry object.

⸻

6.2 Tool: search_agent_entries

Purpose

Search previously created AI entries.

Input

{
  "query": "scroll"
}

Behavior
	•	search only entries with margin_key = "AGENT"
	•	perform case-insensitive substring search
	•	match against content

Response

Returns an array of full entry objects.

Example:

[
  {
    "id": "c421",
    "timestamp": "2026-03-11T10:21:13-05:00",
    "margin_key": "AGENT",
    "bullet": "•",
    "content": "Investigate scroll jitter in TUI",
    "priority": false,
    "pinned": false
  }
]


⸻

6.3 Tool: list_recent_agent_entries

Purpose

Provide recent AI-generated entries for context.

Input

{
  "limit": 10
}

Defaults

limit = 10
max_limit = 50

Behavior

Return the most recent entries where:

margin_key == "AGENT"

Response

Returns an array of entry objects.

⸻

7. System Architecture

7.1 Integration Strategy

The MCP server will be embedded inside the Rapide codebase rather than distributed as a separate binary.

To maintain Rapide's dependency-minimal philosophy, the MCP implementation will use a native stdio transport handler for JSON-RPC 2.0 messages rather than pulling in external MCP libraries.

Benefits:
	•	zero new dependencies
	•	full control over the protocol implementation
	•	reuse of journal storage logic
	•	shared configuration
	•	reduced maintenance
	•	simpler distribution

⸻

7.2 Proposed Project Structure

internal/
  mcp/
    server.go
    registration.go
    types.go

  mcp/tools/
    add_entry.go
    search_agent_entries.go
    list_recent_agent_entries.go

  mcp/adapter/
    journal_adapter.go

  mcp/errors/
    errors.go


⸻

8. Adapter Layer

The MCP layer must not directly manipulate journal files.

Instead it will interact through a dedicated adapter.

Example interface:

type JournalAdapter interface {
    AddAgentEntry(ctx context.Context, content string) (*Entry, error)
    SearchAgentEntries(ctx context.Context, query string, limit int) ([]Entry, error)
    ListRecentAgentEntries(ctx context.Context, limit int) ([]Entry, error)
}

This layer enforces:
	•	AGENT-only scope
	•	entry defaults
	•	safe journal access

⸻

9. Error Handling

Errors should be:
	•	deterministic
	•	user-friendly
	•	protocol-safe

Example errors:

content is required
query is required
limit must be between 1 and 50
unable to append AGENT entry
unable to read AGENT entries

Internal stack traces should never be exposed through MCP responses.

⸻

10. Testing Strategy

Unit Tests

Verify:
	•	validation rules
	•	AGENT-only enforcement
	•	correct default field values
	•	search matching behavior

Integration Tests

Test the MCP server via stdio:
	1.	start MCP server
	2.	call add_entry
	3.	call list_recent_agent_entries
	4.	call search_agent_entries

Verify journal data integrity.

⸻

11. Security Model

The MVP security model is intentionally simple.

Restrictions include:
	•	no network listeners
	•	no remote clients
	•	no access outside AGENT
	•	no file system access beyond Rapide journal storage
	•	no mutation beyond append-only entries

This protects user privacy and keeps AI interactions predictable.

⸻

12. Rollout Plan

Phase 1 — Experimental
	•	hidden command
	•	three MCP tools
	•	validate with local AI tools
	•	minimal documentation

Phase 2 — Real Usage

Use during Rapide development to evaluate:
	•	usefulness
	•	friction points
	•	journaling ergonomics

Phase 3 — Expansion (Optional)

Possible future improvements:
	•	edit/toggle entry states
	•	richer search
	•	multiple AI collections
	•	optional MCP resources
	•	prompt templates
	•	agent identity metadata

These will only be considered if the MVP proves valuable.

⸻

13. Acceptance Criteria

The feature is considered complete when:
	1.	rapide mcp start launches a working MCP server
	2.	AI tools can append entries via add_entry
	3.	entries are always written to AGENT
	4.	search returns only AGENT entries
	5.	search is case-insensitive substring matching
	6.	recent entries can be listed
	7.	MCP cannot access other collections
	8.	journal integrity is preserved
	9.	code remains modular enough for possible extraction

⸻

14. Implementation Order

Recommended development sequence:
	1.	create MCP package structure
	2.	implement MCP server bootstrap
	3.	create journal adapter
	4.	implement add_entry
	5.	implement list_recent_agent_entries
	6.	implement search_agent_entries
	7.	add unit tests
	8.	add integration tests
	9.	validate with a local MCP client

⸻

15. Summary

The Rapide MCP feature provides a small, safe integration point allowing AI tools to use Rapide as a rapid-log capture system.

The MVP intentionally limits scope to:
	•	append-only AI entries
	•	AGENT collection isolation
	•	simple substring search
	•	local stdio communication

This keeps the feature aligned with Rapide’s philosophy while enabling powerful AI-assisted workflows.
